# Standard-Harness工程架构设计

# 一、总体架构：
<img width="2371" height="1494" alt="whiteboard_exported_image (8)" src="https://github.com/user-attachments/assets/b73f4f57-48d9-41f9-8112-1b5eef9e1077" />

## 设计目标：

可承载多个Agent、支持规划与工具使用、具备长短期记忆、安全纵深防御、全链路可观测、并支持快速迭代的AI应用平台。

## 总体技术原则：

- 核心与外围分离：模型权重与训练属于模型工程，其余一切（推理封装、提示词、检索、工具、编排、安全、运维）均属Harness。

- 分层解耦：十层架构（L0–L9），每层独立部署、独立演进。

- 协议标准化：对外工具优先使用MCP协议实现语义互操作，但服务发现与治理依靠通用服务网格（Consul/etcd）。

- 状态外置：Agent运行时状态持久化于数据库，服务无状态，可水平扩展。

- 全链路可观测：OpenTelemetry \+ 专用LLM可观测工具，覆盖从请求到模型到工具的每一个步骤。


# 二、分阶段设计：

## （一）、用户交互与体验阶段：

### 1、交互层：

**（1）、职责：**

接收用户/设计者输入，处理数据模态转换、格式规整、可视化设计操作、响应结果处理展示等。

#### （2）、核心功能：

普通用户多模态请求输入；

开发人员编排设计业务流程；

请求响应结果处理。

## （二）、基础设施与模型服务阶段：
<img width="1852" height="1166" alt="whiteboard_exported_image (3)" src="https://github.com/user-attachments/assets/f84d5284-0cf4-4ec9-99be-b2f7b2b88538" />

### 1、模型运行层：

**（1）、职责：**

将裸模型封装为稳定、高可用的推理服务，对上提供统一的OpenAI兼容接口，并实现多模型、多厂商的动态路由、负载均衡与成本控制。

**（2）、工作方式：**

- LiteLLM作为所有模型调用的唯一入口，配置多个“部署”（deployment），每个部署对应一个模型（例如 `gpt-4o`、`claude-3.5-sonnet`、`mixtral-8x22b`）。

- 路由策略：根据请求参数中的 `model` 字段别名路由；也可配置基于成本或延迟的预算感知路由，将简单意图调度到便宜的小模型。

- 自部署模型由vLLM运行在GPU节点上，前置一个轻量FastAPI适配层（若需要格式转换），然后注册到LiteLLM。

- 对所有上游提供统一的流式响应（SSE），支持取消和超时控制。

**（3）、边界：**

不管理模型权重、训练或微调过程；不关心上层业务提示词的内容。只负责“高效、可靠地生成token”。

## （三）、智能体能力构建阶段：
<img width="4725" height="3200" alt="whiteboard_exported_image (4)" src="https://github.com/user-attachments/assets/65b7df7c-1260-4678-ba66-8b130bbeaf99" />

### 1、提示词及上下文工程层：

#### （1）、职责：

管理模型在每一时刻“看到”的全部信息，包括系统指令、工具描述、检索结果、记忆、对话历史，并在严格的Token预算内完成动态组装。这是Harness系统中最精密、最频繁变动的部分。

#### （2）、工作方式：

- 提示词模板版本化存储在Git仓库中，启动时加载，支持热更新。

- 装配逻辑：接收请求后，查询相关记忆（L5）、检索外部知识（L2，可选）、当前工具定义（L3），全部注入模板上下文。

- Token预算控制器：累加所有文本段的token数，若超过模型上下文窗口限制，按优先级自动裁剪：先裁剪远期对话摘要 → 再裁剪检索文档片段 → 仅保留系统指令、最近几轮对话和必要工具描述。

- 最终生成的消息数组直接传递给L0层的LiteLLM调用。

#### （3）、边界：

不负责模型如何理解这些文本；不执行检索或记忆存储本身，只从其他层“拉取”并组装。

#### （4）、核心组件：

### 2、知识检索与索引层：

#### （1）、职责：

在需要时将外部知识库中的信息按需注入上下文，而不改变模型参数。适用于频繁更新或海量文档的场景。

#### （2）、工作方式：

- 离线：文档通过摄取管道进入，分块、嵌入、写入Milvus。

- 在线：用户问题先经过查询改写（例如将模糊问题分解为多个具体查询），执行混合检索，召回Top\-K块，用Cross\-encoder精排，取前N个片段交给L1的上下文装配器。

- 支持多知识库和定时增量更新。

#### （3）、边界：

不训练嵌入模型或重排序模型；不负责将检索结果直接作为回复，只提供素材给L1。

#### （4）、核心组件：

### 3、工具与函数调用层：

#### （1）、职责：

让Agent安全、可控地与外部世界交互（调用API、执行代码、查询数据库等），负责工具注册、解析、执行、错误处理全流程。

#### （2）、关键设计：MCP 与 Consul 的分工

- MCP 负责工具“说什么语言”：新开发或可改造的工具，优先实现为MCP Server，提供标准的 `tools/list`、`tools/call` 接口，让Agent能自动发现并理解工具签名。

- Consul 负责“工具实例在哪里”：所有工具实例（无论MCP还是HTTP）启动时向Consul注册，附带标签 `protocol: mcp` 或 `protocol: rest`。

- Tool Executor 从 Consul 查询健康实例列表，对MCP工具建立MCP客户端连接；对传统API发出HTTP请求。

- 这种分层设计既获得了MCP的AI原生互操作性，又获得了Consul提供的生产级服务健康、故障转移和负载均衡能力。

#### （3）、工作方式：

- 工具开发人员定义工具Schema（JSON Schema格式），并将其挂载到工具注册中心。

- Agent需要调用工具时，编排层（L4）发出指令，Tool Executor解析后执行，并捕获异常。

- 异常和返回结果被序列化为自然语言消息，注入回对话上下文，让模型能够自我纠正或重新规划。

#### （4）、边界：

不负责模型“是否选择调用工具”（那是L4的职责），也不负责工具内部的业务逻辑实现。

#### （5）、核心组件：

### 4、记忆与用户画像层：

#### （1）、职责：

管理整个Agent的生命周期：推理循环、任务规划、多步骤执行、异常处理、多Agent协作和人机交互。这是将基础LLM能力转化为目标驱动行为的核心引擎。

#### （2）、支持的Agent模式：

- ReAct循环：`思考` → `行动` → `观察` → 循环，直到产生最终回答。

- Plan\-Execute模式：模型首先生成分步计划（任务列表），然后由解释器逐步执行，执行结果可触发重新规划。

- 多Agent协作：通过LangGraph的嵌套图（子图）或并行节点实现。例如“主编Agent”分解任务，将子任务委派给“研究员Agent”和“代码Agent”，并汇总结果。

- 人机协同：在敏感操作（如转账、发送邮件）前通过 `interrupt_before` 暂停图执行，等待人工审批；审批通过后通过API恢复。

#### （3）、关键机制：

- 终止条件：最大步数、超时、成本熔断、检测到无进展循环（同一工具连续失败N次）时强制终止。

- 错误处理：工具调用失败时，将错误信息转换并注入上下文，给模型自我修复的机会；模型无响应时启动降级规则引擎。

#### （4）、边界：

编排层不处理具体的提示词填充（L1）、工具实现（L3）或安全校验（L6），它只负责“何时调用谁”。

#### （5）、核心组件：

## （四）、智能体运行时与编排阶段：
<img width="2375" height="1211" alt="whiteboard_exported_image (5)" src="https://github.com/user-attachments/assets/2259a015-c642-4292-b5de-5c892074832e" />


### 1、控制流与编排层：

#### （1）、职责：

为Agent提供跨会话的记忆能力，记住用户偏好、关键事实和任务进度，使交互个性化和连续化。

#### （2）、记忆生命周期：

- 提取：对话后异步运行抽取流水线，用小模型生成三元组（主体、关系、客体）或偏好摘要，生成嵌入存入Milvus。

- 召回：每次新对话，将当前问题嵌入后搜索Milvus，结合PostgreSQL中的画像，作为L1的素材之一。

- 遗忘：Milvus支持TTL自动过期；用户删除请求通过统一API清除所有存储中的相关数据。

- 过程记忆（未来）：记录Agent“如何成功完成某类任务”的流程模板，可存入向量库或图数据库，用于指导未来规划。

#### （3）、边界：

记忆层只提供记忆的存取和查询，不直接生成回复内容。

## （五）、保障、治理与演进阶段：
<img width="4145" height="3293" alt="whiteboard_exported_image (6)" src="https://github.com/user-attachments/assets/546d8dce-30e9-4514-adb1-5a144c2043d1" />


### 1、安全合规与护栏层：

#### （1）、职责：

构建纵深防御体系，在请求和响应的每个环节防止提示注入、有害内容、数据泄漏和越权操作，确保系统安全合规。

#### （2）、工作方式：

- 请求进入系统后，先经过Kong网关认证和粗粒度过滤，再由Guard Service进行细粒度清洗。

- 模型输出后，同样由Guard Service校验，通过后才返回用户。

- 所有拒绝和修改操作都记录审计日志。

#### （3）、边界：

安全层不修改核心业务逻辑，仅在数据进出关键边界时介入。

#### （4）、核心组件：

### 2、评估与测试层：

#### （1）、职责：

量化Agent回答的质量，建立离线/在线评估体系，驱动持续改进。

#### （2）、工作方式：

- 从生产日志中采样并人工标注构建黄金数据集。

- 离线评估作为CI门禁；在线评估用于监控质量漂移。

- Langfuse提供统一面板，将反馈、评分和Trace关联，快速定位Bad Case。

#### （3）、核心组件：

### 3、可观测与运维层：

#### （1）、职责：

提供对系统健康度、成本、质量和行为的全方面监控，支撑排错和优化。

#### （2）、工作方式：

- 每个服务在启动时注册OpenTelemetry，自动插桩。

- Langfuse与LangGraph/LangChain集成，所有模型调用自动上报。

- Grafana面板展示全局延迟、成本、TOP工具错误、安全拦截统计等。

- 排错时，从Grafana或Langfuse点击一个Trace，可看到从用户请求到工具调用的完整过程。

#### （3）、核心组件：

### 4、开发工具与生命周期管理：

#### （1）、职责：

管理Agent系统的配置、实验、发布全生命周期，实现快速迭代和可靠部署。

#### （2）、工作方式：

- 开发人员修改提示词模板或Agent配置，提交PR。

- CI自动运行Ragas评估，若评分无显著下降则允许合并。

- 合并后触发Docker镜像构建，更新部署清单。

- ArgoCD检测到Git变更，自动滚动更新服务。

- 如需灰度发布，通过Argo Rollouts配置多版本并行。
